## Project Overview

**ShopLogic** is a web-based fleet maintenance management system designed for Class 8 diesel truck and trailer repair shops. Shared floor devices use a **two-step model**: a **staff member** signs in on the main screen (full Supabase session), then the app enters **kiosk mode** so technicians can tap in with **name + PIN** for fast clock-in/out. Managers get real-time visibility into fleet operations; reporting grows in later phases.

### Primary Goals
- **Technician productivity**: Ultra-fast job selection & clock-in/out via kiosk mode (after the device is unlocked by staff)
- **Manager visibility**: Realtime job status, tech allocation, and fleet health
- **Business intelligence**: Reports on labor utilization, parts costs, maintenance schedules, and tech performance
- **Operational efficiency**: Sequence management, standardized job templates, and customizable workflows

### Target Users
- **Technicians**: Field workers clocking into jobs, logging work details
- **Shop managers**: Assign jobs, decline/reschedule, monitor fleet, manage staff
- **Shop owners/admins**: Configure sequences, set labor rates, define PM schedules, view fleet analytics

## Tech Stack & Rationale

### Frontend
- **Framework**: React 18+ (Vite for build tooling)
- **Routing**: TanStack Router (lightweight, modular)
- **State Management**: React Context + Supabase realtime subscriptions (avoid Redux for MVP simplicity)
- **UI Components**: Custom React components (no heavy UI library initially; focus on speed)
- **Styling**: Tailwind CSS (rapid development, consistent theming)
- **Forms**: React Hook Form + Zod validation (lightweight, type-safe)

**Why**: Vite provides fast HMR. React Context handles auth & shared state efficiently. Supabase realtime eliminates polling. Tailwind + custom components keep bundle size small.

### Backend
- **Database & Auth**: Supabase (PostgreSQL + Auth)
  - Built-in JWT authentication (no custom auth service needed)
  - Realtime subscriptions for live job updates
  - Row-level security (RLS) for multi-tenant scalability
  - Free tier serves MVP well; scales to paid tier as needed
- **API**: Supabase Edge Functions (serverless, no separate backend to deploy)
  - VIN decoder service integration
  - Sequence number generation (work order, tech ID, PO)
  - Report calculations aggregation
  - Email notifications (Phase 3)

### Hosting
- **Frontend**: Netlify (auto-deploy on git push, fast CDN, environment variables easy)
- **Database**: Supabase (managed PostgreSQL, realtime subscriptions)

### Development Tools
- **Local Development**: npm + Docker (optional Supabase local dev)
- **Version Control**: Git (GitHub or GitLab)
- **Testing**: Vitest (unit), Playwright (E2E)
- **Linting**: ESLint + Prettier

### Why This Stack
1. **Minimal backend effort**: Supabase Auth + Edge Functions eliminate custom auth service
2. **Realtime by default**: Supabase subscriptions keep techs & managers in sync
3. **Fast deployment**: Netlify auto-deploys; no DevOps complexity for MVP
4. **Type safety**: React + TypeScript throughout
5. **Cost-effective**: Free tiers cover early-stage volume; scales linearly

## Development Phases

### Phase 1: MVP (Weeks 1-8)
**Market-ready core for immediate deployment**

**In scope:**
- Kiosk login (all techs on screen, tap name + optional PIN)
- Customer & vehicle management (CRUD + VIN decoder)
- Job creation with auto-numbering & decline tracking
- Tech dashboard (open work orders → select & clock in/out)
- Employee management (labor rates, last day worked, time clock audit)
- Robust settings panel (sequences, markups, labor rates)
- Realtime tech connection & status
- Basic spreadsheet view (jobs list with editable columns)
- Role-based dashboards (tech vs manager/admin)

**Out of scope (defer to Phase 2):**
- Preventive maintenance tracking & scheduling
- Advanced reporting (paid vs billed, idle time analysis, PM due)
- Canned jobs library
- Parts inventory module
- Customizable email templates
- Batch PO generation

### Phase 2: Fleet Operations & Reporting (Weeks 9-16)
- PM tracking & scheduling
- PM Upcoming + Declined Jobs report
- Tech Paid vs Billed report
- Idle Time report with efficiency insights
- Canned jobs library
- Parts inventory tracking
- Spreadsheet column customization (add/remove/reorder)
- Category-based parts/sublet markup matrix

### Phase 3+: Advanced Features
- Batch PO generation & delivery
- Customizable email templates with placeholders
- Advanced form builder (inspection forms, job sheets)
- Job history & audit trails
- RFID badge integration (upgrade from PIN)
- Mobile app (React Native or Flutter)
- Offline capability for field techs

## Core Flows & Features

### 1. Kiosk Login Flow
**Actor**: Technician (on shared shop floor device)

```
1. Kiosk screen loads
   → Display all ACTIVE technicians (name only, large tap targets)
   
2. Tech taps their name
   → If PIN required: Show PIN entry pad
   → Otherwise: Skip to step 4
   
3. Tech enters optional PIN
   → Validate PIN against users.pin
   → If invalid: Show error, return to step 1
   
4. Session established
   → Store JWT in localStorage
   → Connect to Supabase realtime
   → Redirect to Tech Dashboard
   
5. Idle timeout
   → Track last touch/activity (5-min auto-logout on shared device)
   → Show warning at 4 min: "Time until logout"
   → Auto-logout at 5 min, return to kiosk
```

**Key tables involved:**
- `users` (filter by role='tech' AND active=true)
- `job_logs` (for realtime updates)

**Context**: Realized via `AuthContext` + `KioskLoginScreen.tsx`

---

### 2. Tech Dashboard & Job Selection
**Actor**: Technician (logged in via kiosk)

```
1. Tech sees dashboard
   → List of OPEN work orders assigned to them
   → Columns: Job #, Vehicle VIN, Customer, Description, Status, Time On Job
   
2. Tech selects a job
   → Tap/click job row
   → System creates job_logs entry with clock_in = NOW
   → Show "Clocked In" confirmation + timer
   
3. Tech is now clocked in
   → Can add notes, work category, parts used (future)
   → Timer shows elapsed time on this job
   
4. Tech clocks out
   → Tap "End Job" button
   → System updates job_logs: clock_out = NOW, calculates elapsed time
   → Return to open jobs list (step 1)
```

**Key tables involved:**
- `jobs` (filter by tech_id & status='open')
- `job_logs` (create entry, update clock_out)

**Context**: Realized via `TechDashboard.tsx` + `JobSelector.tsx` + `TechLoggingService.ts`

---

### 3. Manager Job Assignment & Decline Tracking
**Actor**: Shop Manager

```
1. Manager views all jobs (spreadsheet)
   → Columns: Job #, Vehicle, Customer, Assigned Tech, Status, Created Date
   
2. Manager assigns job to tech
   → Edit job row OR create new job
   → Select tech from dropdown
   → Save
   
3. Manager declines a job (optional)
   → Right-click or "Decline" button on job row
   → Popup: "Decline Reason?" (dropdown: Parts unavail, Customer postponed, Equipment issue, Other)
   → Save: Set job.is_declined=true, decline_reason=selection
   → Job disappears from tech's open list (realtime)
   
4. Declined jobs visible in reports (Phase 2)
   → PM Upcoming + Declined Jobs report
   → Groups by decline reason for analysis
```

**Key tables involved:**
- `jobs` (update status, is_declined, decline_reason)

**Context**: Realized via `JobList.tsx` + `JobService.ts`

---

### 4. Settings Panel (Admin)
**Actor**: Shop Owner/Admin

```
WORK ORDER SEQUENCE:
1. Navigate to Settings → Sequences
2. Edit "Work Order Format"
   → Template: WO-{YYYY}-{SEQUENCE}
   → Starting #: 1001
   → Preview: "WO-2025-1001", "WO-2025-1002" (shows next 5)
3. Save → Auto-applied to all new jobs

LABOR RATES:
1. Navigate to Settings → Labor Rates
2. Edit "Default Labor Rate": $85/hour
3. Per-customer override:
   → Customer dropdown
   → Override rate: $95/hour (replaces default for that customer's jobs)
4. Save → Applied to job_logs when clocking out

PARTS/SUBLET MARKUP:
1. Navigate to Settings → Markup Matrix
2. Add row: Type=Parts, Markup%=25
3. Add row: Type=Sublet, Markup%=20
4. Future: Category support (Electrical=30%, Hydraulics=25%)
5. Save → Applied to job costing (Phase 2)
```

**Key tables involved:**
- `settings` (JSON config for sequences, labor rates)
- `customers` (labor_rate_override field)
- `markup_matrix` (rows for each markup rule)

**Context**: Realized via `SettingsDashboard.tsx` + `SequenceSettings.tsx` + `MarkupMatrixSettings.tsx`

---

### 5. Employee Management (Admin)
**Actor**: Shop Owner/Admin

```
1. Navigate to Employees
   → Single screen showing all techs in a table
   → Columns: Tech ID, Name, Role, Labor Rate, Last Day Worked, Last Clock In/Out, Status, Actions
   
2. Edit existing employee
   → Click "Edit" row
   → Modal/form opens: Name, Labor Rate, Last Day Worked, Active/Inactive
   → Save → Updates users table
   
3. Time clock audit
   → "Last Clock In/Out" column shows most recent clock_in + clock_out from job_logs
   → Click to see full clock history (modal with timestamp log)
   
4. Add new employee
   → "Add Employee" button at top
   → Form: Name, Role, Labor Rate, PIN (optional)
   → Save → Creates new users entry
   
5. Deactivate employee
   → Set Active=false
   → Tech disappears from kiosk login (filtered by active=true)
   → Tech's existing job_logs remain for reporting
```

**Key tables involved:**
- `users` (CRUD, labor_rate, last_day_worked, active)
- `job_logs` (for clock history)

**Context**: Realized via `EmployeeList.tsx` + `EmployeeForm.tsx` + `EmployeeService.ts`

---

### 6. VIN Decoder Integration
**Actor**: Admin (creating vehicle)

```
1. Navigate to Vehicles → Add Vehicle
2. Enter VIN (e.g., "1HSCM21606A123456")
3. Click "Decode VIN"
   → Calls Supabase Edge Function: vehicles-vin-decoder
   → Edge Function calls NHTSA API: https://vpic.nhtsa.dot.gov/api/vehicles/DecodeVin/{vin}
   → Returns: Make, Model, Year, Body Type, Gross Vehicle Weight
4. Form auto-populates with decoded data
5. Admin can override fields if needed
6. Save → Vehicle created with decoded details
```

**Key integration:**
- Edge Function: `supabase/functions/vin-decoder.ts`
- Calls: NHTSA API (free, no key required)
- Error handling: If VIN invalid or API down, show error and allow manual entry

**Context**: Realized via `VehicleForm.tsx` + `VINDecoder.ts` + Edge Function

## Supabase Schema Overview

See [/docs/SCHEMA.md](./docs/SCHEMA.md) for full entity relationships and field definitions. Below is a brief summary:

### Core Tables (Phase 1 MVP)

**users**
- `user_id` (UUID, PK)
- `name` (text)
- `role` (enum: 'tech', 'manager', 'admin')
- `pin` (text, nullable; optional 4-6 digit PIN)
- `labor_rate` (money; default $75/hour)
- `last_day_worked` (date, nullable)
- `active` (boolean; default true; filters from kiosk if false)
- `created_at`, `updated_at`

**customers**
- `customer_id` (UUID, PK)
- `name` (text)
- `contact_person` (text)
- `email` (text)
- `phone` (text)
- `address` (text)
- `labor_rate_override` (money, nullable; use instead of default if set)
- `created_at`, `updated_at`

**vehicles**
- `vehicle_id` (UUID, PK)
- `customer_id` (FK → customers)
- `vin` (text, unique)
- `make` (text; from VIN decoder)
- `model` (text)
- `year` (integer)
- `body_type` (text; e.g., "Tractor Unit", "Trailer")
- `gvwr` (integer, nullable; from VIN decoder)
- `mileage` (integer, nullable; current odometer)
- `status` (enum: 'active', 'inactive', 'sold')
- `created_at`, `updated_at`

**jobs**
- `job_id` (UUID, PK)
- `job_number` (text, unique; auto-generated via sequence, e.g., "WO-2025-1001")
- `vehicle_id` (FK → vehicles)
- `tech_id` (FK → users)
- `description` (text)
- `status` (enum: 'open', 'in-progress', 'completed', 'declined')
- `is_declined` (boolean; default false)
- `decline_reason` (text, nullable; e.g., "Parts unavailable", "Customer postponed")
- `estimated_cost` (money, nullable)
- `created_at`, `updated_at`

**job_logs**
- `log_id` (UUID, PK)
- `job_id` (FK → jobs)
- `tech_id` (FK → users)
- `clock_in` (timestamp)
- `clock_out` (timestamp, nullable)
- `elapsed_minutes` (integer, nullable; auto-calculated at clock_out)
- `idle_minutes` (integer, nullable; calculated gap from prior job)
- `notes` (text, nullable)
- `work_category` (text, nullable; e.g., "Diagnostics", "Engine Work", "Electrical")
- `created_at`, `updated_at`

**settings**
- `setting_id` (UUID, PK)
- `config` (JSONB)
  - `default_labor_rate` (number)
  - `work_order_sequence` (object: { format: string, next_number: int })
  - `tech_id_sequence` (object: { format: string, next_number: int })
  - `po_sequence` (object: { format: string, next_number: int })
- `created_at`, `updated_at`

**markup_matrix**
- `matrix_id` (UUID, PK)
- `type` (enum: 'parts', 'sublet')
- `category` (text, nullable; future: "Electrical", "Hydraulics", "Drivetrain")
- `markup_percent` (number; e.g., 25.0)
- `active` (boolean; default true)
- `created_at`, `updated_at`

### Phase 2+ Tables (Defer)
- `canned_jobs` (job templates)
- `pm_schedule` (preventive maintenance rules)
- `parts` (parts inventory)
- `job_costs` (parts + labor + sublet breakdown per job)

## Module Development Guide

Each feature module is semi-independent. Follow this pattern:

### Module Structure
```
modules/[feature]/
├── [Feature]List.tsx          # List/table view
├── [Feature]Detail.tsx        # Read-only detail or modal
├── [Feature]Form.tsx          # Create/edit form
├── [Feature]Service.ts        # Business logic (CRUD, calculations)
├── types.ts                   # Domain models
├── hooks.ts                   # Custom React hooks (queries, mutations)
└── constants.ts               # Local constants (enum values, etc.)
```

### Development Steps for Each Module

1. **Define types** (`types.ts`)
   - Domain models matching Supabase schema
   - Form input types
   - API response shapes

2. **Write Service layer** (`[Feature]Service.ts`)
   - CRUD operations using Supabase client
   - Business logic unrelated to React
   - Pure functions for calculations
   - Error handling

3. **Create custom hooks** (`hooks.ts`)
   - `useQuery(id)` — Fetch single record
   - `useList()` — Fetch & paginate records
   - `useCreate()` — Insert + refetch list
   - `useUpdate(id)` — Update + realtime refresh
   - `useDelete(id)` — Soft or hard delete
   - Use Supabase client + React hooks only

4. **Build UI components** (`[Feature]List.tsx`, `[Feature]Form.tsx`, etc.)
   - Consume custom hooks
   - Keep business logic in Service, not in JSX
   - Use shared components (Button, Input, Table, Modal, etc.)
   - Use Tailwind for styling

5. **Write tests**
   - Unit tests on Service layer
   - Integration tests on hooks (mock Supabase)
   - E2E tests on critical workflows

### Example: Customer Module

**types.ts**
```typescript
export interface Customer {
  customer_id: string;
  name: string;
  email: string;
  labor_rate_override?: number;
  created_at: string;
}

export interface CustomerFormInput {
  name: string;
  email: string;
  labor_rate_override?: number;
}
```

**CustomerService.ts**
```typescript
export async function getCustomer(id: string) {
  const { data, error } = await supabase
    .from('customers')
    .select('*')
    .eq('customer_id', id)
    .single();
  if (error) throw error;
  return data;
}

export async function createCustomer(input: CustomerFormInput) {
  const { data, error } = await supabase
    .from('customers')
    .insert([input])
    .select()
    .single();
  if (error) throw error;
  return data;
}
```

**hooks.ts**
```typescript
export function useCustomer(id: string) {
  const [customer, setCustomer] = useState<Customer | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    getCustomer(id)
      .then(setCustomer)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [id]);

  return { customer, loading, error };
}

export function useCreateCustomer() {
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  const create = async (input: CustomerFormInput) => {
    setLoading(true);
    try {
      return await createCustomer(input);
    } catch (err) {
      setError(err as Error);
      throw err;
    } finally {
      setLoading(false);
    }
  };

  return { create, loading, error };
}
```

**CustomerForm.tsx**
```typescript
export function CustomerForm({ onSave }: { onSave: (c: Customer) => void }) {
  const { create, loading, error } = useCreateCustomer();
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    const customer = await create({ name, email });
    onSave(customer);
  };

  return (
    <form onSubmit={handleSubmit}>
      <Input label="Name" value={name} onChange={(e) => setName(e.target.value)} />
      <Input label="Email" value={email} onChange={(e) => setEmail(e.target.value)} />
      <Button type="submit" disabled={loading}>
        {loading ? 'Saving...' : 'Save'}
      </Button>
      {error && <p className="text-red-500">{error.message}</p>}
    </form>
  );
}
```

## Code Standards & Conventions

### TypeScript
- **Always** use types; never use `any`
- Use `interface` for public contracts, `type` for internal types
- Export types from `types.ts` files
- Use discriminated unions for variant types

### React
- Functional components only (no class components)
- Custom hooks for reusable logic
- Lift state to Context only if 3+ components share it
- Keep components small & focused (max ~200 lines JSX)

### Naming
- Components: PascalCase (`CustomerForm.tsx`)
- Hooks: `use*` prefix (`useCustomer.ts`)
- Services: `*Service.ts` or `*` suffix (`CustomerService.ts`)
- Files: kebab-case for directories, PascalCase for components

### Imports
- Group imports: external → core → modules → shared → types
- Use path aliases (configured in `vite.config.ts`): `@/core`, `@/modules`

### Error Handling
- Service layer: Throw errors
- Hooks/Components: Catch and set state
- User-facing messages: Localize later; use clear English for now
- Log errors to console in dev; send to monitoring in prod (Phase 2+)

### Performance
- Memoize expensive computations: `useMemo`
- Memoize callback functions passed to children: `useCallback`
- Lazy-load routes: `React.lazy` + `Suspense`
- Avoid unnecessary re-renders: Check `key` prop on lists

### Styling
- Tailwind utilities only; no CSS files initially
- Use design tokens: Colors defined in `tailwind.config.js`
- Dark mode support in tailwind config (optional Phase 2+)

### Testing
- Write tests alongside code, not after
- Unit: Service layer logic
- Integration: Hooks + mocked Supabase
- E2E: Critical paths (login → job → clock out)

## Authentication & Authorization

### Overview
- **Auth method**: Supabase Auth (email optional; name + PIN for kiosk)
- **Session storage**: JWT in localStorage
- **Authorization**: Role-based access control (RBAC) via `users.role`
- **Realtime**: Supabase subscriptions update all clients when jobs change

### Login Flow
1. Kiosk displays all active techs
2. Tech taps name → system finds user by name
3. If `users.pin` exists, tech enters PIN (validated client-side for speed; can add server-side check if needed)
4. System calls Supabase Auth to create session OR creates custom JWT
5. Store JWT in localStorage
6. Init Supabase subscription filters

### Authorization Rules (Row-Level Security)

**Tech users can:**
- View own jobs & job_logs
- Update own job_logs (clock_in, clock_out)
- View own user profile

**Manager users can:**
- View all jobs
- Assign jobs to techs
- Decline jobs + set decline_reason
- View all job_logs (for reports)
- Edit settings & markups (initially; Phase 2 restricts to admin only)
- View employee roster

**Admin users can:**
- All manager permissions +
- Full CRUD on users, customers, vehicles, settings, markup_matrix
- Create/delete backup/restore

### Session Management
- JWT expires after 24 hours (configurable in `.env`)
- On logout: Clear localStorage + redirect to kiosk
- On idle (5 mins on kiosk): Auto-logout
- Refresh: Token auto-refreshes on Supabase operations (built-in)

## Key Technical Patterns

### 1. Realtime Job Updates
**Problem**: Manager declines job; tech should see it disappear from their list instantly.

**Solution**: Supabase realtime subscriptions
```typescript
useEffect(() => {
  const subscription = supabase
    .from('jobs')
    .on('*', (payload) => {
      // Payload includes old vs new record
      if (payload.new.is_declined) {
        // Remove from list
      }
    })
    .subscribe();

  return () => subscription.unsubscribe();
}, []);
```

### 2. Auto-Incrementing Sequences
**Problem**: Generate next work order number (WO-2025-1001, 1002, etc.)

**Solution**: Supabase Edge Function + settings table
```typescript
// Edge Function: supabase/functions/generate-sequence.ts
const { next_number } = await supabase
  .from('settings')
  .select('config->>work_order_sequence')
  .single();

const job_number = `WO-${year}-${next_number}`;

await supabase
  .from('settings')
  .update({
    config: {
      ...config,
      work_order_sequence: { ...config.work_order_sequence, next_number: next_number + 1 }
    }
  });
```

### 3. Customizable Spreadsheet Table
**Problem**: Manager wants to show/hide columns, reorder them, inline-edit cells.

**Solution**: React Context + local state for column visibility + Tailwind grid
```typescript
interface TableConfig {
  visibleColumns: string[];
  columnOrder: string[];
}

const TableContext = createContext<TableConfig>({
  visibleColumns: ['name', 'email', 'status'],
  columnOrder: ['name', 'email', 'status']
});

export function Table({ rows, allColumns }: { rows: any[], allColumns: string[] }) {
  const { visibleColumns } = useContext(TableContext);
  return (
    <div className="grid gap-1" style={{ gridTemplateColumns: `repeat(${visibleColumns.length}, 1fr)` }}>
      {visibleColumns.map((col) => (
        <div key={col}>{col}</div>
      ))}
      {rows.map((row) => (
        visibleColumns.map((col) => (
          <div key={`${row.id}-${col}`}>{row[col]}</div>
        ))
      ))}
    </div>
  );
}
```

### 4. Idle Time Calculation
**Problem**: Report tech idle time (gap between jobs).

**Solution**: Query prior job_log > calculate gap
```typescript
export function getIdleMinutes(currentClockIn: Date, techId: string) {
  const priorLog = await supabase
    .from('job_logs')
    .select('clock_out')
    .eq('tech_id', techId)
    .order('clock_out', { ascending: false })
    .limit(1)
    .single();

  if (!priorLog || !priorLog.clock_out) return 0;
  return Math.floor((currentClockIn - new Date(priorLog.clock_out)) / (1000 * 60));
}
```

### 5. Labor Rate Resolution
**Problem**: Get labor rate for tech on job (default vs customer override).

**Solution**: Query cascade
```typescript
export async function getApplicableLaborRate(jobId: string, techId: string) {
  const job = await supabase
    .from('jobs')
    .select('vehicle_id, customers(labor_rate_override)')
    .eq('job_id', jobId)
    .single();

  const customer = job.customers;
  if (customer?.labor_rate_override) {
    return customer.labor_rate_override;
  }

  const tech = await supabase
    .from('users')
    .select('labor_rate')
    .eq('user_id', techId)
    .single();

  return tech.labor_rate || 75; // Default fallback
}
```
